## The trajectory (in order)

### Phase 1 — Docs site pipeline

Operator wanted a working GH Pages preview before doing anything else.
Decisions:
- **MkDocs Material** over Jekyll/Docusaurus/Hugo — markdown-native, matches
  the existing `docs/` layout, strict mode catches broken refs.
- **Modern Pages deploy** via `actions/deploy-pages` (artifact upload),
  not the legacy `gh-pages` branch.
- **Strict build on PR** so broken refs never reach `main`.
- Layered Makefile target pattern (`docs-install` / `docs-serve` /
  `docs-build` / `docs-validate` / `docs-clean`) — sets up the same
  pattern that `templates/new-project/` would adopt.

Mid-build, the dev server showed a confusing two-nav UI (tabs + bold
sections). Operator: "left nav makes no sense". Dropped
`navigation.tabs` + `navigation.sections` from theme features →
single hierarchical left nav. Saved as a feedback memory.

Three commits: docs pipeline, dashboard hardening (next phase), single-nav
collapse.

### Phase 2 — DGX terminal dashboard recipe hardening

The genesis session had left an orphaned 454-line ops doc at
`docs/dgx-terminal-dashboard.md` (full tmux + mosh + nvitop setup). Caught
by the strict-build nav-warning. Four-part hardening:

1. **Templated all PII/identifiers** — operator email, tailnet name
   (`tail6d0ed4`), hostname (`dgx-llm-1`, `spark-2c14`), SSH key
   filename, tailnet IP, ACL tag. Public MIT-licensed repo; the genesis
   convention is templated placeholders.
2. **Moved to `docs/recipes/`** — established the recipes category for
   future ops snippets.
3. **Added header metadata** (Date / Status / Reach) matching genesis style.
4. **Slimmed upstream cheat sheets** to keys actually used (tmux 18→7,
   nvitop 11→6, btop 5→4, ctop 8→4).

### Phase 4 — Recipes (two pairs)

Operator wanted two operational recipes next:

  transition. Placeholder compose paths.
- **`recipes/observability-boot.md`** — 6-step walkthrough from
  "stack templated in repo" to "metrics in Grafana Explore". Credential
  acquisition, .env setup, boot, verify (with dashboard IDs 1860 /
  12239 / 17296 / 893), image-tag pinning post-boot, troubleshooting.

Plus two more recipes (later in the session, but listed here for
recipe completeness):
- **`recipes/token-management-lean-ctx-rtk.md`** — operator's
  daily-driver lean-ctx (MCP) + RTK (Bash hook) pair. 60-99% token
  savings without changing how invocations look. Reconstructed from
  `~/.claude.json` + `~/.claude/settings.json`.
- **`recipes/chrome-devtools-mcp-agent-loop.md`** — Chrome DevTools MCP

  (headless+isolated vs visible+persistent) with when-to-use guidance,
  three concrete loop patterns (visual / network / console-error).
  Modeled on operator's actual use in `orrery` and `podcast_scraper-FUTURE`.

## What is *not* in this v0.2 (deliberately)

- **Claude Code `templates/claude-code/`** — operator's
  `~/.claude/settings.json` is 127K of operator-specific config.
  Sanitization cost > value; load-bearing pieces (RTK hook + lean-ctx
  MCP + chrome-devtools per-project) live in the two MCP recipes.
- **`examples/multi-provider-router/`** — abstraction shape lives in
  podcast_scraper-FUTURE's already-battle-tested provider classes; not
  extracted because genericizing it cleanly is more work than the
  derived value here.
- **`examples/batch-pattern/`** — Anthropic Batch / OpenAI Batch
  patterns are documented in `cloud-ai-workflow.md` text but not
  given a working code skeleton. Yet.
- **Live runs of the recipes.** The `observability-boot`,
  `gpu-mode-swap`, and `bakeoff` recipes are all drafted but the
  operator's first live runs happen post-this-session.
